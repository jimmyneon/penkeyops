# Session and Shift Behavior Documentation

## Start Day and End Day - System Actions

**Start Day** and **End Day** are **system actions**, NOT editable tasks in templates.

### How They Work

1. **Start Day (Start Opening)**
   - Appears automatically when a session exists but `started_at` is NULL
   - Action type: `'start_opening'`
   - When clicked: Updates `shift_sessions.started_at` to current timestamp
   - Cannot be edited in admin - it's generated by the resolver function
   - No template_item exists for this action

2. **End Day**
   - Appears automatically when `can_end_day(session_id)` returns TRUE
   - Action type: `'end_day'`
   - Condition: All required and critical tasks are complete
   - When clicked: Triggers EndOfDayModal with performance summary
   - Cannot be edited in admin - it's generated by the resolver function
   - No template_item exists for this action

### Database Function: `resolve_now_action`

The resolver checks in this order:
1. If session not started → return `'start_opening'` system action
2. If `can_end_day()` returns true → return `'end_day'` system action
3. Otherwise → return next pending task (excluding any End Day template items)

### Template Items Named "End Day"

Any template_items with titles like "End Day", "End of Day", "Confirm End" are:
- **Excluded from the resolver** (filtered out in all queries)
- **Should be deleted** - they're legacy and cause confusion
- The system action replaces them entirely

## Session Auto-Creation

### Daily Session Behavior

Sessions are **auto-created per day**, not reset at midnight.

1. **How It Works**
   - When user loads the app, `useShiftSession` hook checks for today's session
   - Query: `started_at::DATE = CURRENT_DATE AND is_complete = false`
   - If no session exists for today → auto-creates new session
   - Session includes: `site_id`, `started_at` (date only, time set on Start Day click), `shift_type = 'daily'`

2. **No Midnight Reset**
   - Sessions don't reset at midnight
   - Each day gets a new session when first accessed
   - Previous day's sessions remain in database with `is_complete = true`

3. **Session Completion**
   - When End Day is clicked → `is_complete = true`, `completed_at = NOW()`
   - Completed sessions are filtered out by the hook
   - Next day, new session auto-creates

## Tasks After End Day

**Tasks cannot appear after End Day is completed.**

### How This Works

1. **Session Filtering**
   - `useShiftSession` filters: `is_complete = false`
   - Once session marked complete, it's no longer the "active session"
   - UI shows no session → no tasks can load

2. **Resolver Protection**
   - Even if someone tried to query a completed session
   - The resolver would still work, but UI won't call it
   - New session auto-creates for next day

3. **Template Instantiation**
   - Templates are instantiated when session is created
   - Checklist instances and results created at session start
   - Completed sessions don't get new task instances

## Admin Template Editor

**Start Day and End Day do NOT appear in admin template editor.**

- They are system actions generated by `resolve_now_action` function
- No template_items exist for them in the database
- Admin can only edit actual task template_items
- If any "End Day" template_items exist, they should be deleted (they're excluded from resolver anyway)

## Summary

| Feature | Behavior |
|---------|----------|
| Start Day | System action, appears when session not started |
| End Day | System action, appears when all required/critical tasks done |
| Session Creation | Auto-created daily when user first accesses app |
| Midnight Reset | No - new session created on first access next day |
| Tasks After End Day | Impossible - completed sessions filtered out |
| Admin Editing | Start/End Day cannot be edited (they're not template items) |
| End Day Template Items | Should be deleted - they're legacy and excluded from resolver |
